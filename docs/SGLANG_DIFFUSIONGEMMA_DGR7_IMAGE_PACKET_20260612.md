# SGLang DiffusionGemma DG-R7 Image Smoke Packet

Date: 2026-06-12 JST

Objective: run the first live image prompt gate after the static vision-warning
audit. This is a narrow multimodal smoke, not an image benchmark.

## Scope

- Model: `google/diffusiongemma-26B-A4B-it`
- Runtime: stock upstream DiffusionGemma policy
- Attention/KV: Triton attention, BF16/auto KV, eager/no graphs
- API: OpenAI `/v1/chat/completions` with `image_url` content
- Images: synthetic PNGs generated by
  `scripts/diffusion_gemma_image_quality_client.py`

Out of scope:

- FlashInfer VO-split
- NVFP4 or mixed-KV
- capacity and throughput
- broad image understanding quality

## Gate

Run:

```bash
bash scripts/run_sglang_dgemma_dgr7_image_smoke.sh
```

Expected pass:

- server reaches readiness under the stock DiffusionGemma launch
- logs prove the stock Triton DiffusionGemma policy
- two repeats of the red/blue image mention both red and blue
- two repeats of the green-square image mention green
- outputs are non-empty and byte-stable per image prompt

Prompt policy: use short descriptive prompts, not terse "answer with one word"
forms. DG-R2 already showed DiffusionGemma can denoise overly constrained terse
prompts to empty output even when broader direct prompts are semantically fine.

Stop on red:

1. readiness failure
2. OpenAI multimodal request parsing failure
3. empty or unstable responses
4. semantically wrong color answer

## Interpretation

GREEN only means the real image processor plus vision-forward path is live for
a small deterministic color-recognition smoke. It does not turn prior text-only
DiffusionGemma rows into broad multimodal quality claims.
